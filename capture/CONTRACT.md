# Контракт capture-сервиса (заморожен 2026-06-12)

Рендерер программируется строго по этому документу. Изменения — только
синхронно с обеих сторон и с обновлением этого файла.

## Видео — WebRTC

- Сигналинг: `POST /offer` тело `{sdp, type}` → ответ `{sdp, type}`.
- Кадр: side-by-side двойной ширины `[H, 2W]`, rgb24:
  левая половина — RGB, правая — альфа, продублированная в 3 канала
  (живёт в люме — YUV 4:2:0 её не портит). Семпловать `.r` правой половины.
- 720p вход → кадр 2560×720. До первого кадра пайплайна трек шлёт
  ЧЁРНЫЙ кадр 2560×720 — рендерер обязан трактовать чёрный/отсутствующий
  поток как IDLE (мастер-спека §3.1).
- Темп: 30 fps (dev, зашит в SbsTrack; прод-60 — подпроект Ops).

## Телеметрия — WS `/ws`, 15 Гц, один тип сообщения

```json
{"type": "presence", "present": bool, "distanceCm": float|null,
 "coverage": float, "bbox": [x0,y0,x1,y1]|null, "errors": int, "fps": float}
```

- `bbox` нормирован 0..1; нижняя кромка = «ноги» для контактной тени (§4).
- `joints` ПОКА НЕТ (появится с ZED, подпроект Ops).

## Health — HTTP GET `/health`

```json
{"ok": bool, "frames": int, "fps": float, "present": bool,
 "errors": int, "lastError": str|null}
```

## Отличия от мастер-спеки §3.1 (осознанные)

- Нет `personCount`; нет отдельных WS-типов `joints`/`health`
  (health — только HTTP; fps едет внутри presence-сообщения).
- Добавлены: `coverage`, `bbox`, `errors`.
Все ключи зафиксированы тестами `capture/tests/test_server.py`.
