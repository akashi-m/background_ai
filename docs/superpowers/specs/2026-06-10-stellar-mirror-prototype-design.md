# Прототип «Зеркало Stellar» — дизайн

**Дата:** 2026-06-10
**Статус:** утверждён
**Контекст:** прототип для интерактивной инсталляции Stellar Residence (City Mall Tashkent). Полное ТЗ заказчика — рекламная конструкция с «зеркалом», в котором посетитель видит себя внутри квартиры. Этот документ описывает **прототип на ноутбуке**, который проверяет главный риск: работает ли «магия зеркала» (сегментация + параллакс) убедительно.

## 1. Цель прототипа

Доказать на ноутбуке (MacBook, встроенная или USB-вебкамера), что:

1. Посетитель видит себя «в интерьере квартиры» — фигура вырезается из видео и вставляется в 3D-интерьер.
2. Интерьер реагирует на положение головы зрителя (параллакс, off-axis проекция) — «отражение» комнаты движется физически правдоподобно, как в настоящем зеркале.
3. Режим «Окно»: переключение на вид из окна (панорама города) с тем же параллаксом.
4. Задержка и качество краёв достаточны, чтобы эффект не разваливался.

Срок: 2–4 недели. Результат показывается заказчику для решения о продакшн-версии.

## 2. Вне скоупа прототипа

- Киоск-режим, watchdog, работа 12 ч/день — продакшн.
- Режим ожидания (звёздное небо), активация по датчику присутствия — продакшн.
- Переключение нескольких вариантов интерьера — в прототипе один интерьер.
- Жесты, тач, автотаймер переключения — в прототипе клавиши `M`/`W`.
- Удалённое обновление контента, LED-подсветка, QR-панели — продакшн/конструкция.
- Реальный контент квартир Stellar Residence — у заказчика пока нет материалов; используем бесплатные ассеты.

## 3. Выбранный подход (и отвергнутые альтернативы)

**Выбрано: «виртуальное зеркало»** — обычный экран + камера + сегментация. Комбо двух режимов: «Зеркало» (главный) и «Окно» (свитч).

Отвергнуто:

- *Полупрозрачное зеркало + дисплей за ним* (буквальная трактовка ТЗ) — физически не позволяет «вставить» человека в интерьер: отражение и картинка дисплея накладываются, интерьер просвечивает сквозь отражение.
- *VR-шлем* — очереди, гигиена, персонал, поломки; убивает walk-up-опыт в ТЦ.
- *Аватар с жестовым управлением* — дорого, зловещая долина, ненадёжные жесты в шумном ТЦ.

**Стек: веб** — Three.js + MediaPipe Tasks Vision, запуск в Chrome. Причины: работает на Маке разработчика, самые быстрые итерации, сегментация и трекинг лица из коробки, демо запускается с любого ноутбука. Путь в продакшн: Chrome-киоск на Windows-машине с GPU, либо порт на Unity/UE при потребности в кино-картинке.

## 4. Архитектура

Один веб-апп (SPA), без бэкенда. Конвейер каждого кадра:

```
Вебкамера 1080p/30fps
  → MediaPipe: маска фигуры (сегментация) + позиция головы (x, y, z из размера лица)
  → One-Euro фильтр (сглаживание позиции головы)
  → Three.js: рендер сцены с off-axis проекцией (экран = плоскость зеркала)
  → Композитор: фон-интерьер + вырезанная фигура сверху, мягкие края, зеркальный флип
  → Экран 60 fps
```

Ключевые решения:

- **Развязка частот:** ML работает на ~30 fps (частота камеры), рендер — на 60 fps с последней готовой маской и интерполированной позицией головы. Параллакс остаётся плавным, даже если сегментация притормаживает.
- **Off-axis проекция** (generalized perspective projection, Kooima): фрустум камеры строится из позиции глаз зрителя и физических углов экрана. Для режима «Зеркало» виртуальная камера ставится в зеркально отражённую позицию зрителя.
- **Фигура зрителя не параллаксится** — она и есть «отражение», зритель двигает её сам. Параллакс применяется только к 3D-фону.
- **Зеркальный флип** по горизонтали для всего кадра в режиме «Зеркало» (отражение, а не видео).

## 5. Структура кода

```
src/
├── tracking/    захват камеры, MediaPipe (сегментация + лицо), One-Euro фильтр
├── render/      Three.js сцена, off-axis камера, композитор (фон + фигура)
├── scenes/      интерьер (GLTF) и вид из окна (панорама), загрузка ассетов
├── app/         стейт-машина режимов ЗЕРКАЛО/ОКНО, переключение клавишами
└── debug/       панель FPS/задержки, визуализация трекинга, калибровка
```

Каждый модуль самостоятелен: `tracking` ничего не знает о Three.js (выдаёт маску-текстуру и сглаженную позицию головы), `render` ничего не знает о MediaPipe (принимает позицию глаз и текстуру маски), `app` соединяет их.

## 6. Режимы

| | ЗЕРКАЛО (главный) | ОКНО |
|---|---|---|
| Контент | 3D-интерьер (GLTF) | фото-панорама города на сфере (слои с глубиной — возможное улучшение, не в прототипе) |
| Фигура зрителя | композитится поверх | скрыта |
| Параллакс | да, от позиции головы | да |
| Переключение | клавиша `M` | клавиша `W` |

Переход между режимами — короткий кроссфейд (< 0,5 с), без перезагрузки сцены (обе держатся в памяти).

## 7. Калибровка

Off-axis проекции нужна физика: ширина/высота видимой области экрана (см) и смещение камеры относительно центра экрана. Маленькая панель настроек (открывается клавишей `C`), значения хранятся в localStorage. Оценка расстояния до зрителя — по межзрачковому расстоянию в кадре (среднее ~63 мм), точность ±10% достаточна для эффекта.

## 8. Обработка сбоев

- **Лицо потеряно** → позиция головы плавно затухает к центральной (нейтральной) точке; при повторном захвате — плавный подхват без рывка (тот же фильтр).
- **Несколько лиц** → трекается самое крупное (= ближайший зритель).
- **Нет доступа к камере** → полноэкранное сообщение с инструкцией, не чёрный экран.
- **Ассеты не загрузились** → сообщение об ошибке с именем файла.

## 9. Контент прототипа

- Интерьер: бесплатная GLTF-сцена жилой комнаты с окном (Sketchfab CC / Poly Haven), подбирается под эстетику жилой недвижимости.
- Вид из окна: бесплатная фото-панорама города.
- Контент лежит в `public/assets/`, пути — в одном конфиге (`scenes/config.ts`), чтобы замена контента на материалы заказчика была тривиальной.

## 10. Производительность и качество (бюджеты)

- Сквозная задержка камера→экран: цель < 150 мс (ощущается как «почти зеркало»).
- Рендер: 60 fps на MacBook (M-серия); сегментация ≥ 24 fps.
- Маска: разрешение модели MediaPipe (256×256) + смягчение краёв в шейдере (feather 2–4 px, лёгкая эрозия), чтобы края не «дышали».
- Debug-панель показывает FPS рендера, FPS сегментации и оценку сквозной задержки — оценка эффекта не на глаз.

## 11. Тестирование

- **Юнит-тесты:** математика off-axis проекции (известные позиции глаз → ожидаемые фрустумы), One-Euro фильтр (сглаживание/отзывчивость), стейт-машина режимов, оценка расстояния по лицу.
- **Ручная проверка («тест магии»):** двигаешь головой влево/вправо/ближе — комната в «отражении» сдвигается правдоподобно; задержка не раздражает; края фигуры приемлемы при комнатном свете; свитч M↔W мгновенный.
- Визуальную убедительность юнит-тестами не поймать — поэтому debug-панель с числами + чек-лист ручной проверки в README.

## 12. Критерии приёмки прототипа

1. На MacBook в Chrome: видишь себя в 3D-интерьере, фон параллаксится от движения головы.
2. Сквозная задержка < 150 мс по debug-панели, рендер стабильно ≥ 50 fps.
3. Режим «Окно» переключается клавишей, кроссфейд < 0,5 с.
4. Потеря лица / несколько людей / отказ камеры не ломают картинку.
5. Замена интерьера = замена файла + правка одного конфига.

## 13. Путь в продакшн (контекст, не обязательства)

Из ТЗ заказчика, на потом: 4K-экран 85–98" вертикально, ПК с GPU, режим ожидания «звёздное небо», активация при приближении на 2–3 м (детекция человека камерой, реакция < 1 с), несколько интерьеров (свитч < 2 с), удалённое обновление контента, киоск-режим 12 ч/день. Прототип архитектурно этому не противоречит: детекция присутствия — то же MediaPipe, смена контента — уже через конфиг, веб-стек переносится на Windows-киоск.
