# Битрейт WebRTC — резкость live-композита — спека

**Дата:** 2026-06-13
**Статус:** утверждён в чате («го»)
**Родитель:** этап «зеркало качественно». Матт уже чистый в `/frame.png`
(lossless), но в live-зеркале мылит WebRTC.

## 1. Узкое место

Дефолтный кодек aiortc 1.14 — VP8 с `MAX_BITRATE=1.5 Мбит/с`,
`DEFAULT_BITRATE=500 кбит/с`. Для SBS-кадра 2560×720 (тем более 3840×1080)
это в разы мало → мыло. Энкодер стартует с DEFAULT и клампит REMB-таргет
в `[MIN, MAX]` — все три модульные глобалы `aiortc.codecs.vpx`,
читаются в момент вызова.

## 2. Решение

Сервер слушает ТОЛЬКО 127.0.0.1 (спека мастера §6) — loopback, реальной
конкуренции за полосу нет, congestion-control не нужен. Прижимаем битрейт
к целевому намертво: `MIN = DEFAULT = MAX = target` до создания энкодера.
Так картинка стартует резкой сразу (без медленного REMB-разгона) и не падает
от заниженных оценок полосы.

- Флаг `--bitrate <Мбит/с>`, дефолт **8** (с запасом для 2K SBS; для 4K на
  проде поднять до 20+). Конфиг-поле `bitrate_mbps`.
- `configure_bitrate(mbps)` в `webrtc.py`: ставит три глобала vpx в
  `int(mbps*1e6)`. Вызывается один раз в `main.py` до `run_app`.
- Кодек оставляем VP8 (без новых зависимостей); H264 не форсим.

## 3. Риск / фолбэк

Если браузер всё равно зажимает (REMB не даёт разгон) — добавить `b=AS:<kbps>`
в m-секцию answer-SDP (мунж в `handle_offer`). Пока не делаем: на loopback
прижатого MIN достаточно. Проверка вживую покажет.

## 4. Тестирование

парсинг `--bitrate` (дефолт 8, валид float). Приёмка: live `/viewer` на S24 —
лицо/текст резкие, не мыло; сравнить с `/frame.png` (должны сойтись).
